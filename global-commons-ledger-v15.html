The Wage Structure

One ladder: earned through upskilling, sized to clear its own debt

Wages aren't flat, and they aren't set independently of what the system owes. Workers move up a skill ladder through the platform's training pipeline — simulation-based modules, sandbox testing, peer verification — and the population share at each tier is chosen so that total payroll, plus Delper's debt service, closes exactly.

The Financing Model

Delper's infrastructure loan

Delper provides the upfront capital to build the automated production network and receives amortized repayment plus interest. Every worker's gross economic output has to cover both their own wage and a share of that debt service — the two numbers together are what "gross value per worker" actually means.

Institutional Underwriting

Financing the loan like sovereign debt

Delper's $675T CapEx position is structured as an institutional Medium-Term Note — sold to central banks, sovereign wealth funds, and pension funds under a joint sovereign guarantee, with the underwriting ratios a credit committee would actually check.

Debt Service Coverage Ratio $31.323T monthly inflow ÷ $3.834T monthly debt service 8.17x

Well above the 1.25x threshold considered investment-grade — the production loop generates 817% of what's needed to clear debt charges monthly, even before accounting for any safety margin.

Operating Cost Ratio $27.489T payroll ÷ $31.323T total production 87.76%

87.76% of everything the system generates returns directly to workers as take-home payroll — the debt siphon claims the remaining 12.24%.

Capital Multiplier $375.88T annualized production ÷ $675T CapEx principal 0.56x / yr

Every dollar of upfront infrastructure investment unlocks $0.56 of physical goods and services annually — 3.90x cumulative return over the 7-year build-out.

Debt-to-Assets Leverage $1,060.75T funded (CapEx + OpEx) ÷ $1,060.75T constructed value 1.00x

Every note issued is matched by physical and digital infrastructure — factories, farms, server complexes, energy fields.

Reading the leverage ratio correctly: the 1.00x figure isn't an independent asset appraisal — it's constructed by defining the built infrastructure's value as equal to what was spent building it (CapEx + one year of OpEx). That's a legitimate accounting identity, not evidence of resale or collateral value on its own; a real credit committee would still want independent appraisal of the physical assets before treating this as a risk-mitigant.

Guaranteed jointly and severally by participating sovereign states — if one nation faces a localized disruption, the remaining guarantors absorb its share automatically, keeping the credit rating insulated from any single country's volatility. Structured to qualify as Basel III/IV High-Quality Liquid Assets, Class 1, with 0% risk-weighting for holding institutions.

Coordination Layer, Automated

The payout engine

A smart contract automates the monthly distribution: it enforces Delper's debt service ahead of any worker payout during the debt era, then automatically pivots into the 50/50 abundance split and R&D routing once the loan clears — with zero administrative override.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.26;

/// Core monthly constants (18-decimal scaled)
uint256 constant MONTHLY_DELPER_DEBT_SERVICE = 3_834_000_000_000 * 10**18;
uint256 constant MONTHLY_CAPEX_PORTION        = 2_870_000_000_000 * 10**18; // amortizes to zero at Year 31
uint256 constant MONTHLY_OPEX_INTEREST_PORTION = 964_000_000_000 * 10**18; // revolving — never fully clears
uint256 constant MONTHLY_BASE_HUMAN_PAYROLL   = 27_489_000_000_000 * 10**18;
uint256 constant MONTHLY_TARGET_CAPACITY     = 31_323_000_000_000 * 10**18;
uint256 constant DEBT_ERA_DURATION = 30 * 365 days;

/// Prioritized macro-siphon — runs once per cycle, before any worker can claim
function executeMacroSiphon() external onlyOracle {
    uint256 bal = GCL_CREDIT_TOKEN.balanceOf(address(this));
    require(bal >= MONTHLY_TARGET_CAPACITY, "GCL: production deficit");

    bool postDebt = block.timestamp >= (SYSTEM_GENESIS_TIMESTAMP + DEBT_ERA_DURATION);

    if (!postDebt) {
        // Years 1-30: Delper repaid first, unconditionally, in full
        require(GCL_CREDIT_TOKEN.transfer(DELPER_TREASURY, MONTHLY_DELPER_DEBT_SERVICE), "debt xfer failed");
        emit MonthlyDebtServiceCleared(MONTHLY_DELPER_DEBT_SERVICE, block.timestamp);
    } else {
        // Year 31+: only the CapEx portion frees up — OpEx interest is revolving, keeps flowing
        require(GCL_CREDIT_TOKEN.transfer(DELPER_TREASURY, MONTHLY_OPEX_INTEREST_PORTION), "opex interest xfer failed");

        uint256 freed = MONTHLY_CAPEX_PORTION;
        uint256 delperShare = freed / 2;
        uint256 userShare   = freed / 2;
        uint256 rnd = (delperShare + userShare) / 2;

        require(GCL_CREDIT_TOKEN.transfer(DELPER_TREASURY, delperShare / 2), "royalty xfer failed");
        require(GCL_CREDIT_TOKEN.transfer(JOINT_RND_POOL, rnd), "R&D xfer failed");
        emit AbundanceDividendTriggered(delperShare, userShare, rnd);
    }
}

/// Worker claims — gated on cycle interval, tier-mapped, upgrades post-Year-31
function claimWorkerWage() external {
    Worker storage w = registry[msg.sender];
    require(w.isActive, "GCL: inactive profile");
    require(block.timestamp >= w.lastPayout + 30 days, "GCL: cycle not reached");

    bool postDebt = block.timestamp >= (SYSTEM_GENESIS_TIMESTAMP + DEBT_ERA_DURATION);
    (uint256 netWage, uint256 credits) = _tierPayout(w.tier, postDebt);

    w.lastPayout = block.timestamp;
    require(GCL_CREDIT_TOKEN.transfer(msg.sender, netWage), "wage xfer failed");
    emit WorkerWagePaid(msg.sender, netWage, credits);
}

During the debt era, Delper is paid first and in full, every cycle. Once Year 31 arrives, only the CapEx amortization portion ($3.188T/mo) clears — the OpEx interest portion ($0.803T/mo) sits on a revolving facility and keeps flowing to Delper indefinitely, funding its continuity royalty alongside the freed CapEx split.

Scaling to 6B Users

Why this runs on an appchain, not mainnet

Individual on-chain transactions for 6 billion monthly wage claims would be computationally infeasible on a public chain. Claims are batched into a single Merkle root per cycle; each worker submits a small proof rather than triggering a database lookup.

At 6B claims/month × 21,500 gas, network overhead runs about $9.03M/month on a dedicated appchain — funded from the system-maintenance budget, with zero gas cost passed to workers.
